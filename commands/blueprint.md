# Blueprint（設計図スキル）

1行の目標を、マルチセッションの実行計画に変換するスキルです。
大きなプロジェクト（新サイト制作、研修プログラム設計、事業計画）で使います。

## いつ使うか
- 1セッションでは終わらない大きなタスク
- 複数の工程が絡み合うプロジェクト
- 失敗のリスクを最小化したい重要な仕事

## 5フェーズ・パイプライン

### Phase 1: RESEARCH（調査）
**目的**: 必要な情報を全て集める

実行手順:
1. ユーザーの目標を明確化（1文に要約）
2. 関連する既存リソースを洗い出し:
   - メモリ内の関連ファイル（~/.claude/projects/memory/）
   - Desktopの関連フォルダ
   - 過去のセッション要約
3. 不足情報の特定（何がわからないか）
4. 必要な調査をリストアップ

**出力**: research-brief.md（調査結果のサマリー）

### Phase 2: DESIGN（設計）
**目的**: 実行計画の骨格を作る

実行手順:
1. 全体を独立した工程に分解（最大10工程）
2. 各工程の:
   - 入力（何が必要か）
   - 出力（何を作るか）
   - 依存関係（何が先に終わっている必要があるか）
   - 推定セッション数
3. クリティカルパス（最も長い依存チェーン）を特定
4. リスクポイントの洗い出し

**出力**: 工程表（テーブル形式、ユーザーが好む形式）

| # | 工程 | 入力 | 出力 | 依存 | セッション数 |
|---|------|------|------|------|------------|

### Phase 3: DRAFT（計画書作成）
**目的**: 各セッションで何をするかの詳細計画

実行手順:
1. 各工程をセッション単位に分割
2. 各セッションの計画書を作成:
   - **Goal**: このセッションで達成すること
   - **Context**: 前のセッションからの引き継ぎ事項
   - **Steps**: 具体的な作業手順（5-10ステップ）
   - **Definition of Done**: 完了条件
   - **Rollback Plan**: うまくいかない場合の代替策
3. セッション間の依存関係を図示

**出力**: セッション別計画書

### Phase 4: REVIEW（レビュー）
**目的**: 計画の穴を見つける（敵対的レビュー）

実行手順:
1. **Red Team**（攻撃者視点）: この計画が失敗するシナリオを3つ挙げる
2. **Blue Team**（防御者視点）: 各失敗シナリオへの対策を考える
3. **Auditor**（監査者視点）: 計画の一貫性・完全性をチェック

チェック項目:
- [ ] 各工程の出力が次の工程の入力を満たしているか
- [ ] 並行して進められる工程が正しく特定されているか
- [ ] 最悪ケースでも部分的な成果が残るか
- [ ] ユーザーの判断が必要なポイントが明確か

### Phase 5: REGISTER（登録）
**目的**: 計画を実行可能な形で保存

実行手順:
1. 最終計画を ~/Desktop/blueprints/{project-name}/ に保存
2. ACTIVE_PROJECTS.md に登録（最大3プロジェクト制限を確認）
3. 最初のセッションの計画をメモリに保存
4. SHARED_TASK_NOTES を初期化:
   - `/tmp/SHARED_TASK_NOTES.md` を作成
   - テンプレート:
     ```
     # Shared Task Notes
     **プロジェクト**: {project-name}
     **作成日時**: {date}
     **目的**: {goal}

     ## 完了したステップ

     ## 現在の状態

     ## 未解決の問題

     ## 決定事項（変更禁止）
     ```
   - 各セッション開始時にこのファイルを読み、終了時に更新する

## アンチパターン（やってはいけないこと）
- 計画段階で実装を始める（計画と実行を混ぜない）
- 1セッションに詰め込みすぎる（1セッション = 1-2工程が上限）
- 「完璧な計画」を目指す（80%の計画で十分。実行しながら調整）
- 依存関係を無視して並列化する

## ユーザーの既存プロジェクト管理との連携
- `~/Desktop/ACTIVE_PROJECTS.md` で最大3プロジェクト管理
- `~/Desktop/PROJECT_COMPLETION_SYSTEM.md` の完了フレームワークと連動
- スコープクリープ防止ルール（CLAUDE.md記載）を尊重

## 使い方の例
ユーザー: 「プロジェクトAの管理画面を作りたい」
→ `/blueprint` が起動
→ 5フェーズで計画書を作成
→ 「全5セッション、推定合計10時間」のような計画が出る
→ ユーザーが承認 → Session 1から実行開始

## 完了後の推奨チェーン
- プロジェクト計画完了 → 最初のセッション開始
- 各セッション完了時 → SHARED_TASK_NOTES.md を更新
- 全セッション完了後 → `/audit harness` で環境チェック
