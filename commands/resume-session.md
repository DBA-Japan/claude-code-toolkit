# セッション復元（Resume Session）

前回 `/save-session` で保存したセッション状態を読み込み、中断した作業の続きから再開するコマンド。

## 使い方

```
/resume-session                    # 最新のセッションファイルを読み込む
/resume-session 2026-03-21         # 指定日付のセッションを読み込む
/resume-session プロジェクトA      # タイトルにキーワードを含むセッションを検索
/resume-session ~/.claude/session-saves/2026-03-21_153000_プロジェクトA修正.md  # 直接パス指定
```

## 引数

$ARGUMENTS

---

## プロセス

### Step 1: セッションファイルを見つける

**引数なしの場合:**
1. `~/.claude/session-saves/` を確認する
2. 最も新しいファイルを選ぶ
3. フォルダが存在しないか、ファイルがない場合:
   ```
   セッションファイルが見つかりません（~/.claude/session-saves/）
   作業を中断するときに /save-session を実行すると、ここにファイルが保存されます。
   ```
   → ここで停止する

**引数ありの場合:**
- **日付（`YYYY-MM-DD`）**: その日付で始まるファイルを検索し、最新のものを選ぶ。複数ある場合は一覧を表示してユーザーに選ばせる
- **キーワード（日付でもパスでもない文字列）**: ファイル名にそのキーワードを含むものを検索。複数ある場合は一覧を表示してユーザーに選ばせる
- **ファイルパス**: そのファイルを直接読み込む
- 見つからない場合は明確に報告して停止する

### Step 2: セッションファイルを完全に読み込む

ファイルの全内容を読む。要約しない。

### Step 3: ブリーフィング（状況報告）を出す

以下のフォーマットで、セッションの状態を報告する:

```
--- セッション復元 ---

ファイル: [ファイルパス]
プロジェクト: [プロジェクト名]
概要: [何をしていたかの要約、2〜3文]

■ 現在の状態:
  完了: [件数] 件
  作業中: [ファイル一覧]
  未着手: [ファイル一覧]

■ 前回失敗したこと（再試行しない）:
  ❌ [失敗した方法] — 理由: [なぜ失敗したか]
  ❌ [失敗した方法] — 理由: [なぜ失敗したか]

■ ブロッカー・未解決の疑問:
  [ブロッカーがあれば記載]

■ 次にやること:
  [セッションファイルに書かれた次の一手]
  [未定の場合: 「次のステップは未定。方針を一緒に決めましょう」]

--- 再開準備完了。何から始めますか？ ---
```

### Step 4: ユーザーの指示を待つ

**ここが最も重要: 勝手に作業を開始しない。ファイルの編集もしない。**

ユーザーが「続けて」「やって」等と言ったら、セッションファイルの「次にやること」に従って作業を開始する。

ユーザーが別の指示を出したら、そちらに従う。

---

## 「前回失敗したこと」の取り扱い

**この項目は復元時に必ず表示する。省略しない。**

復元後の作業中も、セッションファイルに記載された失敗アプローチは避ける:

- 同じ方法を再試行しそうになったら、**必ず警告する**:
  ```
  ⚠️ 前回のセッションで、この方法は失敗しています:
  [失敗した方法] — 理由: [なぜ失敗したか]

  別のアプローチを試しますか？ それとも理由が解消されたので再試行しますか？
  ```
- ユーザーが明示的に「それでもやって」と言った場合のみ再試行する

## エッジケース

### 複数のセッションファイルが見つかった場合

一覧を表示してユーザーに選ばせる:

```
以下のセッションファイルが見つかりました:

1. 2026-03-21_153000_プロジェクトA修正.md
2. 2026-03-21_100000_LP改修.md
3. 2026-03-20_180000_ヒアリングシートv3.md

どれを読み込みますか？（番号で選択）
```

### セッションファイルが7日以上前の場合

警告を表示する:

```
⚠️ このセッションは [N日前] のものです。
ファイルやコードが変更されている可能性があります。
```

その後、通常通りブリーフィングを表示する。

### セッションファイル内のファイルがディスク上に存在しない場合

ブリーフィングに警告を追加する:

```
⚠️ 以下のファイルはセッションで参照されていますが、現在ディスク上に存在しません:
  - path/to/missing-file.ts
```

### セッションファイルが空または壊れている場合

```
セッションファイルが見つかりましたが、内容が空または読み取れません。
/save-session で新しいセッションファイルを作成してください。
```

---

## 絶対ルール

1. **勝手に作業を開始しない。** ブリーフィングを出したら、必ずユーザーの指示を待つ
2. **「前回失敗したこと」を必ず表示する。** たとえ「なし」でも表示する
3. **セッションファイルを変更しない。** 読み取り専用。履歴記録として保持する
4. **同じ失敗を繰り返しそうになったら警告する。** これが `/resume-session` の最大の価値
5. **ファイルの存在確認をする。** セッションファイルで参照されているファイルがまだ存在するか確認する
