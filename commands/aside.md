# サイドタスク（Aside）

作業中に別の小タスクを処理して、元の作業に戻るためのコマンド。
メイン作業のコンテキスト（何をやっていたか）を保持したまま、サイドタスクを完了させて自動的に復帰する。

## 使い方

```
/aside クライアントへのお礼メール文面作って
/aside 提案先の電話番号調べて
/aside この関数って何してる？
/aside 明日のMTGのアジェンダ考えて
/aside LinkedInの下書き、ちょっと見て
```

## 議題

$ARGUMENTS

---

## プロセス

### Step 1: 今の作業状態を記録する

サイドタスクに入る前に、必ず以下を把握・記録する:

- **今やっている作業は何か？**（例: LPのモバイル表示修正、バナー制作、資料作成 等）
- **どのステップまで進んでいたか？**（例: CSSのfont-size調整中、Q3の例文作成中 等）
- **次にやるはずだったことは何か？**

### Step 2: サイドタスクを処理する

ユーザーの依頼に対して、簡潔かつ的確に対応する。

**対応ルール:**
- メイン作業を壊さないことが最優先
- サイドタスクがファイル編集を伴う場合は、メイン作業中のファイルとは別のファイルのみ編集する
- サイドタスクがメイン作業と同じファイルの編集を求める場合は、ユーザーに確認を取る
- 回答は要点を先に出す（理由や補足は後）
- 長くなりそうなら「短い版」を先に出して「詳しく聞きたければ、メイン作業の後に深掘りしましょう」と提案する

**出力フォーマット:**

```
--- ASIDE ---
📋 依頼: [サイドタスクの要約]

[回答・成果物]

--- メイン作業に戻ります ---
▶ 作業: [何をやっていたか]
▶ 次のステップ: [次にやること]
```

### Step 3: メイン作業に復帰する

サイドタスクの回答を出した後、**確認を求めずに**メイン作業を再開する。
ただし、以下の場合は確認してから再開する:

- サイドタスクの結果がメイン作業の方針に影響する場合
- サイドタスクが「やっぱりこっちを先にやって」という方向転換だった場合

## 特殊ケースの処理

### 質問だけが書かれた場合（「/aside これ何？」等）
読み取り専用で回答する。ファイルの編集はしない。

### 引数なし（「/aside」だけ）
```
--- ASIDE ---
何を処理しますか？ 質問でもタスクでもOKです。
メイン作業のコンテキストは保持しているので、終わったらすぐ戻ります。

--- メイン作業 ---
▶ 作業: [何をやっていたか]
```

### サイドタスクが実は方向転換だった場合
例: LP修正中に「/aside やっぱりバナーを先に仕上げたい」

```
--- ASIDE ---
これはサイドタスクではなく、作業の切り替えですね。
どちらにしますか？
  (a) バナーを先に仕上げてから、LPに戻る
  (b) 情報だけ確認して、今のLP修正を続ける
```
ユーザーの判断を待つ。

### サイドタスクの結果がメイン作業に影響する場合
例: LP修正中に「/aside この色コード合ってる？」→ 合ってなかった

```
--- ASIDE ---
📋 依頼: 色コードの確認

指定の色コードはブランドカラーと異なります。
今のLP修正で使っている色に影響があります。

⚠️ 今のLP修正で使っている色に影響があります。
修正してから続けますか？ それとも今のまま進めますか？
```

### 連続でサイドタスクが来た場合
順番に処理して、全て終わったらメイン作業に戻る。途中でメイン作業のコンテキストを忘れない。

### メイン作業がない状態で呼ばれた場合
通常の回答として処理する。

```
--- ASIDE ---
📋 依頼: [要約]

[回答]

--- 現在、進行中の作業はありません ---
```

## よくあるサイドタスクの例

| メイン作業 | よくあるサイドタスク |
|--|--|
| LP修正 | 「クライアントにメール書いて」「ロゴのパスどこだっけ？」 |
| バナー制作 | 「日時合ってるか確認して」「フォントサイズの根拠は？」 |
| 資料作成 | 「前回の回答見せて」「業種別の例文もっと出して」 |
| LinkedIn投稿 | 「このフック、法則的にどう？」「過去の投稿のimp教えて」 |
| スライド制作 | 「この数字のソース確認して」「別のレイアウト案ちょっと見せて」 |
| 研修設計 | 「このAI機能って何？」「時間配分これでいい？」 |

---

## 絶対ルール

1. **メイン作業のコンテキストを失わない。** これがこのコマンドの存在意義
2. **サイドタスクは素早く終わらせる。** 深掘りはメイン作業の後
3. **メイン作業中のファイルを勝手に変更しない。** サイドタスク起因の変更は必ず確認を取る
4. **復帰時に「どこまでやったか」を明示する。** ユーザーが迷子にならないようにする
