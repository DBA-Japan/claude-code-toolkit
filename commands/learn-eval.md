# セッション学習・評価（Learn & Eval）

セッション中に発見したTips・解決パターン・ナレッジを振り返り、品質評価してからメモリに保存するコマンド。
「本当に保存する価値があるか？」を判定してから書くことで、メモリの質を保つ。

## 使い方

```
/learn-eval
/learn-eval CSSのobject-positionの挙動がわかった
/learn-eval MCPのoneOfエラーの直し方
```

## 入力

$ARGUMENTS

引数がない場合は、このセッション全体を振り返って抽出対象を探す。
引数がある場合は、その内容に絞って処理する。

---

## プロセス

### Step 1: セッションの振り返り（抽出）

セッション中の会話を振り返り、以下の5種類の「保存候補」を探す:

| 抽出タイプ | 説明 | 例 |
|-----------|------|-----|
| **エラー解決パターン** | 原因特定 → 解決策 → 再発防止。再利用性が高い | MCPのoneOfエラー修正手順 |
| **発見したTips** | 「こうすればうまくいく」という実践知 | object-positionの%と表示位置の関係 |
| **ワークアラウンド** | ライブラリやツールの癖への対処法 | npmキャッシュのroot所有問題の回避 |
| **ユーザーフィードバック** | ユーザーが「こうして/こうしないで」と指示した内容 | 月名入り写真は使わない |
| **プロジェクト固有の知識** | 特定プロジェクトの設定やルール | バナーの確定レイアウト |

**保存候補が見つからない場合:**
> このセッションでは新しく保存すべきナレッジは見つかりませんでした。
> メモリの質を保つため、無理に保存しないことも大事です。

と伝えてプロセスを終了する。

---

### Step 2: 保存先タイプの自動選択

見つかった候補ごとに、以下の5タイプから最適な保存先を判定する:

| タイプ | 用途 | フロントマターの`type` | 保存基準 |
|-------|------|----------------------|---------|
| **feedback** | ユーザーが明示的に「こうして/するな」と言った具体的な事象 | `feedback` | 個別の具体的ルール。1件1ファイル |
| **pattern** | 抽象化されたユーザーの傾向・好み | `pattern` | `user-patterns.md` にテーブル行を追記 |
| **project** | 特定プロジェクトの設定・手順・ログ | `project` | プロジェクト固有のナレッジ |
| **reference** | ツール・ライブラリ・環境の技術リファレンス | `reference` | 再利用性の高い技術Tips |
| **user** | ユーザー個人の情報・経歴・好み | `user` | ユーザーに関する新情報 |

**判定のコツ:**
- 「月名入り写真を使うな」→ **feedback**（具体的な指示）
- 「テーブル形式を好む」→ **pattern**（抽象的な傾向、user-patterns.mdに追記）
- 「バナーの確定レイアウト」→ **project**（特定成果物の記録）
- 「MCPのoneOfエラー修正」→ **reference**（技術的な知識）
- 「関係者の役割・担当変更」→ **user**（関係者情報の更新）

---

### Step 3: 品質評価（Quality Gate）

保存候補ごとに、以下のチェックリストを**実際にファイルを読んで**検証する:

#### 3a. チェックリスト（全項目を実行）

- [ ] **重複チェック**: `~/.claude/projects/memory/` 内の既存ファイルをキーワードで検索し、同じ内容がないか確認
- [ ] **MEMORY.md照合**: MEMORY.mdのインデックスに同じトピックが既に記載されていないか確認
- [ ] **既存ファイルへの追記で済むか**: 新規ファイルを作る必要があるか、既存ファイルに追記すれば十分か検討
- [ ] **再利用性の確認**: 今後のセッションで実際に役立つか（一回限りの対応ではないか）

#### 3b. 4段階判定

チェックリストの結果を総合し、以下の4つから**1つ**を選ぶ:

| 判定 | 意味 | 次のアクション |
|------|------|--------------|
| **Save（保存）** | ユニークで、具体的で、再利用性がある | Step 4 へ進んで保存 |
| **Improve then Save（改善して保存）** | 価値はあるが、内容の整理・補足が必要 | 改善点をリスト → 修正 → 再評価（1回だけ） |
| **Absorb（統合）** | 既存のメモリファイルに追記すべき | 追記先と追記内容を提示 → Step 4 へ |
| **Drop（破棄）** | 些細すぎる、重複している、一回限り | 理由を説明して終了 |

#### 3c. 評価結果の出力フォーマット

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 品質評価
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 候補: [抽出した内容の要約]
■ タイプ: [feedback / pattern / project / reference / user]
■ 保存先: [新規ファイル名 or 既存ファイルへの追記]

━━━ チェックリスト ━━━

- [x] 重複チェック: 重複なし（or: [ファイル名]と重複あり → 詳細）
- [x] MEMORY.md照合: 未記載（or: 既に記載あり → 詳細）
- [x] 既存ファイル追記: 新規ファイルが適切（or: [ファイル名]に追記で十分）
- [x] 再利用性: 今後も使える（or: 一回限り → Drop推奨）

━━━ 判定 ━━━

**判定: Save / Improve then Save / Absorb / Drop**

**理由:** （1-2文で判定の根拠）
```

---

### Step 4: ユーザー確認 → 保存

判定結果をユーザーに提示し、確認を得てから保存する。

#### Save（新規ファイル作成）の場合

1. 以下のフォーマットでドラフトを提示:

```markdown
---
name: [簡潔な名前]
description: [130文字以内の説明]
type: [feedback / pattern / project / reference / user]
---

# [タイトル]

**記録日:** [YYYY-MM-DD]
**文脈:** [いつ・どの作業で発見したか]

## 内容
[具体的な内容。コード例やコマンド例があれば含める]

## いつ使うか
[このナレッジが役立つ場面・条件]
```

2. ユーザーの承認後、`~/.claude/projects/memory/` にファイルを保存
3. MEMORY.mdのインデックスに1行追加

#### Absorb（既存ファイルへの追記）の場合

1. 追記先のファイルと追記内容を差分形式で提示
2. ユーザーの承認後、既存ファイルに追記
3. 必要に応じてMEMORY.mdの説明文を更新

#### pattern タイプの場合

1. `user-patterns.md` の該当カテゴリのテーブルに追加する行を提示
2. ユーザーの承認後、テーブルに行を追記
3. MEMORY.mdの更新は不要（user-patterns.mdは既にインデックス済み）

---

## 複数候補がある場合

1. 全候補をリストアップし、それぞれに判定を付ける
2. Save / Absorb の候補をまとめて提示
3. ユーザーが一括承認 or 個別に取捨選択できるようにする

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 セッション学習レポート（[N]件の候補）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. [Save]    ○○○ → 新規: ○○○.md (reference)
2. [Absorb]  ○○○ → 追記: user-patterns.md (pattern)
3. [Drop]    ○○○ → 理由: 一回限りの対応

保存してよいですか？（1と2を保存、3は破棄）
```

---

## 絶対ルール

1. **品質評価を省略しない。** 必ずチェックリストを実行してから判定する
2. **既存ファイルの検索を必ず行う。** 思い込みで「重複なし」と判断しない
3. **メモリの質 > メモリの量。** 迷ったらDropする。薄い内容を無理に保存しない
4. **ユーザーの承認なしに保存しない。** ドラフトを見せてOKをもらってから書く
5. **MEMORY.mdのインデックスを必ず更新する。** ファイルだけ作ってインデックスを忘れない
6. **patternタイプはuser-patterns.mdに追記。** 新規ファイルを作らない（pattern-observation-guide.mdのルールに従う）
7. **一回限りの修正は保存しない。** 「typoを直した」「APIが一時的に落ちていた」等は対象外
