# セッション保存（Save Session）

作業の中断時に、現在のセッション状態を構造化して保存するコマンド。
自動保存（hooks）とは別の、詳細な手動保存。重要な作業の中断・再開に使う。

## 使い方

```
/save-session
/save-session プロジェクトA修正
```

## 引数

$ARGUMENTS

---

## プロセス

### Step 1: セッションの内容を整理する

保存する前に、以下を全て把握する:

- **何を作ったか / 何を変更したか**（ファイル一覧と状態）
- **何が動いたか**（成功したこと＋その証拠）
- **何が失敗したか**（失敗した方法＋正確な理由）← **最重要セクション**
- **まだ試していないこと**（候補のアプローチ）
- **次にやるべきこと**（再開時の最初の一手）
- **決定事項**（次回のセッションが蒸し返さないように）

### Step 2: タイトルを決める

引数にタイトルが指定されていれば、それを使う。
指定がなければ、ユーザーに聞く:

```
このセッションにタイトルをつけてください。
例: 「プロジェクトA修正」「LP改修」「ヒアリングシートv3」
```

### Step 3: 保存ディレクトリを確認する

```bash
mkdir -p ~/.claude/session-saves
```

### Step 4: セッションファイルを作成する

ファイル名: `~/.claude/session-saves/YYYY-MM-DD_HHMMSS_[タイトル].md`

例: `2026-03-21_153000_プロジェクトA修正.md`

### Step 5: 以下のフォーマットで書き込む

全セクションを正直に書く。該当なしの場合は「なし」と書く。セクションを省略しない。
不完全なファイルは、正直に「なし」と書いたセクションがあるファイルよりも有害。

---

## セッションファイルのフォーマット

```markdown
# セッション: [タイトル]

**日付**: YYYY-MM-DD
**時刻**: HH:MM頃
**プロジェクト / 作業場所**: [プロジェクト名やディレクトリパス]
**概要**: [このセッションで何をしていたか、1行で]

---

## 何を作っていたか（背景・目的）

[1〜3段落で、このセッションの作業内容を説明する。
このセッションの記憶がゼロの状態で読んでも理解できるレベルで書く。
何をしていたか、なぜそれが必要か、全体のどこに位置する作業か。]

---

## 成功したこと（証拠つき）

[確認済みのことだけ書く。各項目に「なぜ動くと分かるか」の証拠を添える。
証拠がないものは「未確認」セクションに移す。]

- **[成功した内容]** — 確認方法: [具体的な証拠]
- **[成功した内容]** — 確認方法: [具体的な証拠]

該当なしの場合: 「まだ確認済みの成功項目はない」

---

## 失敗したこと（理由つき）← 最重要セクション

[試したけど失敗した全てのアプローチを書く。
各失敗に対して「なぜ失敗したか」を正確に書く。
次回のセッションが同じ失敗を繰り返さないために、これが最も重要。
「うまくいかなかった」は不十分。「Xのエラーが出た。原因はY」のレベルで書く。]

- **[試した方法]** — 失敗理由: [正確なエラーメッセージや原因]
- **[試した方法]** — 失敗理由: [正確なエラーメッセージや原因]

該当なしの場合: 「失敗した試行はまだない」

---

## まだ試していないこと

[有望だがまだ試していないアプローチ。会話の中で出たアイデア。
次回のセッションが何を試すべきか分かるよう、具体的に書く。]

- [アプローチ / アイデア]
- [アプローチ / アイデア]

該当なしの場合: 「特定の未試行アプローチはない」

---

## ファイルの状態

[このセッションで触ったファイル全て。各ファイルの状態を正確に記録する。]

| ファイル | 状態 | 備考 |
|---------|------|------|
| `パス/ファイル名` | 完了 | [何をするファイルか] |
| `パス/ファイル名` | 作業中 | [何が終わって、何が残っているか] |
| `パス/ファイル名` | 壊れている | [何が壊れているか] |
| `パス/ファイル名` | 未着手 | [予定だが未着手] |

該当なしの場合: 「このセッションでファイルの変更はなかった」

---

## 決定したこと

[アーキテクチャの選択、トレードオフ、選んだアプローチとその理由。
次回のセッションが同じ議論を繰り返さないために記録する。]

- **[決定内容]** — 理由: [なぜこれを選んだか]

該当なしの場合: 「重要な意思決定はなかった」

---

## ブロッカー・未解決の疑問

[解決できていない問題。答えが出なかった質問。待っている外部依存。]

- [ブロッカー / 疑問]

該当なしの場合: 「ブロッカーなし」

---

## 次にやること（再開時の最初の一手）

[再開時に何も考えずに着手できるレベルで、次の一手を具体的に書く。]

[次の一手が不明な場合: 「次のステップは未定。『まだ試していないこと』と『ブロッカー』を確認してから方針を決める」]

---

## 環境メモ（該当する場合のみ）

[プロジェクトの実行に必要なコマンド、環境変数、起動すべきサービスなど。
標準的なセットアップの場合はこのセクションごと省略してよい。]
```

### Step 6: ユーザーに確認する

ファイルを書き込んだ後、内容を表示して確認を求める:

```
セッションを保存しました: [ファイルパス]

内容を確認してください。修正や追記があれば教えてください。
```

ユーザーの修正要望があれば反映する。

---

## 自動保存との違い

| | 自動保存（hooks） | `/save-session`（このコマンド） |
|---|---|---|
| **タイミング** | セッション終了時に自動 | ユーザーが手動で呼ぶ |
| **保存先** | `~/.claude/session-summaries/` | `~/.claude/session-saves/` |
| **詳細度** | 軽い要約 | 構造化された詳細記録 |
| **用途** | ログ・履歴 | 重要な中断の復元 |
| **復元** | 参照のみ | `/resume-session` で完全復元 |

---

## 絶対ルール

1. **「失敗したこと」セクションを絶対に省略しない。** 次のセッションが同じ失敗を繰り返す最大の原因はこのセクションの欠落
2. **証拠のない「成功」を成功セクションに書かない。** 「たぶん動いている」は成功ではない
3. **全セクションを書く。** 該当なしなら「なし」と書く。セクション自体を省略しない
4. **ユーザーの確認を得てから完了とする**
5. **タイトルは必ずつける。** 後で `/resume-session` で一覧表示するときに識別できるようにする
