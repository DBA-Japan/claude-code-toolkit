# 自分のスキルを作る方法

## スキルとは

Claude Codeのスキル（カスタムコマンド）は、`~/.claude/commands/` ディレクトリに `.md` ファイルを置くだけで使えるようになる「定型プロンプトの自動化」です。

```
~/.claude/commands/
├── context.md         ← /context で呼び出せる
├── audit.md           ← /audit で呼び出せる
├── my-skill.md        ← /my-skill で呼び出せる
└── ...
```

ファイル名がそのままコマンド名になります（拡張子 `.md` は除く）。インストール・設定は不要で、ファイルを置いた瞬間から使えます。

---

## スキルの基本構造

スキルは通常のMarkdownファイルですが、いくつかの特別な記法が使えます。

### $ARGUMENTS（引数の受け取り）

```markdown
# /my-skill のスキル定義

ユーザーの入力: $ARGUMENTS

以下の手順で処理してください:
1. ...
2. ...
```

`/my-skill これを分析して` と呼び出すと、`$ARGUMENTS` が「これを分析して」に置き換えられます。引数がなくても動くように設計するのが基本です。

---

### モード判定パターン

引数によって動作を切り替えるパターンです。

```markdown
$ARGUMENTS が空の場合 → デフォルトモード（最もよく使う動作）を実行
$ARGUMENTS が "詳細" を含む場合 → 詳細モードを実行
$ARGUMENTS が "リセット" の場合 → リセット処理を実行
```

このパターンを使うと、1つのスキルで複数の用途をカバーできます（後述のハブパターン）。

---

### ステップ構造

スキルはステップ番号で構造化すると読みやすく、Claudeも迷いません。

```markdown
## Step 1: 現状確認
[やること]

## Step 2: 分析
[やること]

## Step 3: 出力
以下の形式で出力してください:
- 項目A:
- 項目B:
```

---

## 良いスキルの特徴

### 1. 1つのスキルに1つの目的

悪い例: `/analyze-and-fix-and-report` — 分析・修正・レポート生成を1つに詰め込む
良い例: `/analyze`（分析のみ）→ 完了後に `/fix` を提案

スキルは「1つのことをうまくやる」ように設計してください。

### 2. 手順が明確

Claudeは曖昧な指示でも何かしようとしますが、スキルでは具体的な手順を書くほど品質が安定します。

```markdown
# 曖昧（避ける）
ファイルを確認して問題があれば直してください。

# 明確（推奨）
## Step 1: 対象ファイルの確認
glob パターン `**/*.md` で全Markdownファイルを列挙する。

## Step 2: 問題の検出
各ファイルで以下をチェックする:
- [ ] 見出しレベルが正しいか（H1 → H2 → H3 の順）
- [ ] リンク切れがないか

## Step 3: 修正
問題があったファイルを Edit ツールで修正する。
```

### 3. 出力フォーマットを定義する

```markdown
## 出力形式
結果は以下のテーブルで返してください:

| ファイル名 | 問題の種類 | 修正内容 |
|-----------|-----------|---------|
| ...       | ...       | ...     |

最後に「合計X件の問題を修正しました」と1行で締めてください。
```

出力形式を定義することで、スキルの結果が毎回一貫した形式になります。

## テンプレート: 汎用スキルの雛形

以下をコピーして `~/.claude/commands/` に保存してください。

```markdown
# /スキル名 — 一言で何をするスキルか

## 概要
このスキルは [目的] を達成するためのスキルです。

## 使い方
- `/スキル名` — デフォルト動作
- `/スキル名 詳細` — 詳細モード
- `/スキル名 ヘルプ` — このスキルの使い方を表示

---

## 処理フロー

$ARGUMENTS が "ヘルプ" の場合:
このスキルの使い方を表示して終了。

$ARGUMENTS が空または通常の入力の場合:
以下のステップを実行する。

### Step 1: 準備
[具体的な処理を記述]

### Step 2: 実行
[具体的な処理を記述]

### Step 3: 出力
以下の形式で結果を返す:

**結果サマリー**
- [項目]: [値]

**詳細**
[必要に応じて詳細を展開]
```

---

## ハブパターン: 複数モードを1コマンドに統合

関連するスキルが複数ある場合、1つのハブスキルに統合すると管理が楽になります。

### 例: `/context` ハブ

`/context-budget`、`/context-compact`、`/context-optimize` という3つのスキルを `/context` 1つに統合した例:

```markdown
# /context — コンテキスト管理ハブ

$ARGUMENTS を以下のルールで自動判定する:

- 空 または "budget" → Budget モード（診断）
- "compact" または "今compact?" → Compact モード（判断）
- "optimize" または "重い" → Optimize モード（治療）

---

## Budget モード（デフォルト）
現在のコンテキスト消費状況を診断する。
[処理の詳細...]

## Compact モード
今 /compact すべきかを判断する。
[処理の詳細...]

## Optimize モード
コンテキストを削減するアクションを実行する。
[処理の詳細...]
```

**ハブパターンのメリット:**
- コマンド名を覚える量が減る
- 関連スキル同士の文脈が共有される
- 判定ロジックをClaude に任せられる

## スキルチェーン: 完了後に次を提案するパターン

あるスキルが完了した後、自然な次のステップを提案する仕組みです。

```markdown
### Step 4: 完了後の提案
処理が完了したら、以下を提案してください:

「分析が完了しました。次のステップとして:
- `/fix` — 検出した問題を自動修正
- `/report` — 結果をレポート形式で出力

どちらを実行しますか？」
```

CLAUDE.mdにスキルチェーンのテーブルを書いておくと、すべてのスキルに自動的に適用されます。

---

## アンチパターン: やってはいけないこと

### スキルが大きくなりすぎる

1つのスキルファイルが500行を超えてきたら分割のサインです。スキルが長いほど、Claudeが途中でどのステップにいるか混乱しやすくなります。

### 曖昧な手順

「適切に処理してください」「最適化してください」のような指示は避けてください。Claudeは指示通りに何かしますが、毎回異なる結果になります。

### 出力形式が未定義

出力形式を指定しないと、回答のたびにフォーマットが変わります。スキルは「再現性」が命なので、必ず出力形式を定義してください。

### 副作用が不透明

「実行する」と書くだけでなく、「何をするか」「何を変更するか」「元に戻せるか」を明記してください。特にファイル操作・ネットワーク通信を伴うスキルでは重要です。

## スキルのインストール方法（外部スキル）

aitmpl.com や agentskills.io で見つけたスキルは、以下のコマンドで追加できます（claude-peers等のパッケージが対応している場合）:

```bash
npx skills add <author/repo>
```

または手動で:

```bash
# スキルのMarkdownをダウンロードして commands/ に配置
curl -o ~/.claude/commands/skill-name.md https://raw.githubusercontent.com/...
```
