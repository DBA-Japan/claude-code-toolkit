# メモリシステムの設計思想

## なぜメモリが必要か

Claude Codeのセッションは「揮発性」です。セッションを閉じると、会話中に積み上げた文脈・決定・発見はすべて消えます。次のセッションで同じ説明を繰り返し、同じ議論をして、同じ結論に至る——これは非効率です。

メモリシステムは「セッションをまたぐ知識の蓄積」を実現します。Claudeが毎回読み込む場所にファイルを置くことで、前回の文脈を引き継いだ状態でセッションを再開できます。

---

## MEMORY.md の役割

`~/.claude/projects/{プロジェクトパス}/memory/MEMORY.md` はメモリシステムの中核です。

### MEMORY.md はインデックスである

MEMORY.mdは「何がどこに書いてあるか」を示す目次ファイルです。詳細は別ファイルに切り出し、MEMORY.mdからリンクします。

```markdown
# Memory Index

## プロジェクトA
- [詳細設計](./project-a-design.md) — アーキテクチャの決定経緯
- [API仕様](./project-a-api.md) — エンドポイント一覧

## Tips
- [CSS Tips](./css-tips.md) — 間違えやすいCSS挙動まとめ
```

### 200行制限の理由

MEMORY.mdはClaude Codeが**毎セッション自動読み込み**します。長ければ長いほどコンテキストを消費します。200行を超えたら詳細を別ファイルに移動してください。

---

## 4つのメモリタイプ

### 1. user（ユーザー情報・好み）

**何を書くか:** あなた自身に関する情報。Claudeに「この人はどういう人か」を教えるためのファイル。

**例:**
```markdown
---
type: user
topic: preferences
updated: 2026-01-15
---

## コミュニケーションスタイル
- 結論から先に話してほしい
- 専門用語は避けて平易な言葉で
- 提案は必ず理由とセットで

## 作業の癖
- 「直して」と言ったらすぐ実行、確認不要
- スクリーンショットを添付した場合はその箇所を特定してから修正
```

**保存先の例:** `memory/user-preferences.md`

---

### 2. feedback（フィードバック・修正指示）

**何を書くか:** 「これは嫌だった」「これが良かった」という体験的な知識。同じミスを繰り返させないためのファイル。

**例:**
```markdown
---
type: feedback
topic: web-design
updated: 2026-02-20
---

## やってはいけないこと
- カードレイアウトを均一に並べる → バラバラな高さ・サイズが好み
- マウスカーソルを丸い形に変える → デフォルトのまま
- モバイルで letter-spacing を大きくする → 改行崩れの原因

## 好きなデザインパターン
- 斜めのリボンが交差するビジュアル
- 白ベースにアクセントカラー
```

**保存先の例:** `memory/feedback-web-design.md`

---

### 3. project（プロジェクト固有の知識）

**何を書くか:** 特定のプロジェクトに関する決定・仕様・状態。「このプロジェクトの現在地」を示すファイル。

**例:**
```markdown
---
type: project
topic: website-rebuild
updated: 2026-03-01
---

## 現在の状態
- Heroセクション: 完成・デプロイ済み
- Aboutセクション: 未着手
- デプロイ先: Netlify（URL: xxx）

## 技術的決定
- フレームワーク: Vanilla JS（フレームワークなし）
- アニメーション: GSAP + ScrollTrigger
- 日本語フォント: Noto Sans JP

## 残課題
- [ ] モバイルでのナビゲーション修正
- [ ] OGP設定
```

**保存先の例:** `memory/project-website.md`

---

### 4. reference（参考資料・知識ベース）

**何を書くか:** 繰り返し参照する技術情報・手順書・チェックリスト。「辞書」として機能するファイル。

**例:**
```markdown
---
type: reference
topic: deployment
updated: 2026-01-10
---

## Netlify デプロイ手順
1. `npm run build` でビルド
2. `netlify deploy --prod --dir=dist`
3. デプロイ後に `/` でレスポンスを確認

## よくあるエラーと対処
- `Build failed`: Node.js バージョン確認（.nvmrc を確認）
- `404 on refresh`: `_redirects` ファイルに `/* /index.html 200` を追加
```

**保存先の例:** `memory/reference-deployment.md`

---

## frontmatter の書き方

各メモリファイルの先頭に frontmatter を書くと、後から整理しやすくなります。

```markdown
---
type: user | feedback | project | reference
topic: ファイルの主題（1〜3語）
updated: YYYY-MM-DD
confidence: low | medium | high （情報の確実性）
---
```

必須ではありませんが、ファイルが増えてきたときに「これは何のファイルか」を素早く判断できます。

## 保存すべきもの / 保存すべきでないもの

### 保存すべきもの

- **再現可能な手順:** 「○○をするときはこの手順」
- **好み・フィードバック:** 「これが良かった/嫌だった」の体験知
- **重要な決定とその理由:** 「なぜAではなくBを選んだか」
- **よくある失敗と対処法:** 「このエラーはこう直す」
- **プロジェクトの現在地:** 「どこまで進んでいるか」

### 保存すべきでないもの

- **一時的な情報:** 今日だけ使う内容
- **コードの全文:** コードはファイルに存在する。メモリに書かなくてよい
- **会話のログ:** 会話全体を写すのではなく、教訓・結論だけを抽出
- **すぐ変わる情報:** バージョン番号、日付、一時的なURL

## メモリの整理・メンテナンス

### 月1回の棚卸し

定期的に以下をチェックします:

1. **古い情報の確認:** 3ヶ月以上更新されていないファイルは内容が古い可能性
2. **重複の排除:** 同じ内容が複数のファイルに分散していないか
3. **MEMORY.mdの行数確認:** 200行を超えていたら詳細を別ファイルに移動
4. **使われていないファイル:** MEMORY.mdからリンクされていないファイルは削除候補

### `/learn-eval` との連携

セッション終了前に `/learn-eval` を使うと、学んだことを整理してからメモリに保存できます。

```
/learn-eval
```

4段階の評価で仕分けします:
- **Save:** そのままメモリに保存
- **Improve:** 表現を改善してから保存
- **Absorb:** 今回は参考にしたが保存不要
- **Drop:** 間違いや不要な情報、削除

---

## よくある失敗

### メモリの肥大化

「とりあえず全部保存しよう」とすると、MEMORY.mdがすぐに200行を超えます。重要度の低い情報を保存しすぎると、本当に重要な情報が埋もれます。

**対処:** 保存前に「これは1ヶ月後も役立つか?」と自問してください。

### 重複

同じ内容を複数のファイルに書いてしまうと、どちらが最新か分からなくなります。

**対処:** 新しく書く前に既存ファイルを検索してください。`/learn-eval` はこの重複チェックを自動で行います。

### 古い情報の放置

「以前はAという方法が正解だったが、今はBが正解」という状態でも、古いメモリが読まれ続けます。

**対処:** frontmatter の `updated` 日付を確認し、古いファイルには `[廃止]` プレフィックスをつけて削除候補に。

### ファイルを作りすぎる

メモリファイルが50個を超えると管理が大変になります。「別ファイルにするか、既存ファイルに追記するか」は常に既存ファイルへの追記を優先してください。

## メモリシステムの全体像

```
MEMORY.md（インデックス、200行以内）
    ├── user-preferences.md    ← ユーザーの好み・スタイル
    ├── feedback-design.md     ← デザインフィードバック
    ├── project-A.md           ← プロジェクトAの現在地
    ├── project-B.md           ← プロジェクトBの現在地
    └── reference-tools.md     ← ツール・手順書
```

新しいセッション開始時、Claude Codeは自動的にこれらのファイルを読み込み、前回の文脈を引き継いだ状態で作業を再開します。
