# コンテキスト管理の完全ガイド

## コンテキストウィンドウとは

Claude Codeが「一度に読める量」の上限をコンテキストウィンドウといいます。人間の作業机に例えると、コンテキストウィンドウは机の広さです。広ければ多くの資料を広げて作業できますが、机がいっぱいになると新しい資料を置くために古い資料を片付けなければなりません。

会話が長くなる、読み込むファイルが増える、MCPツールを多く有効にする——これらがすべてコンテキストを消費します。

---

## モデル別の上限

| モデル | コンテキストウィンドウ上限 |
|--------|--------------------------|
| Claude Opus | 1,000,000 トークン（1M） |
| Claude Sonnet | 200,000 トークン（200K） |
| Claude Haiku | 200,000 トークン（200K） |

※ トークンの目安: 日本語1文字 ≒ 1〜3トークン、英語1単語 ≒ 1〜2トークン

Sonnet・Haiku で長時間作業する場合は特に注意が必要です。

---

## 固定消費: 何が常にコンテキストを使っているか

セッション開始時点から、以下のものが「固定費」としてコンテキストを消費しています。

| 項目 | 消費の目安 | 説明 |
|------|-----------|------|
| CLAUDE.md | 内容による（数百〜数千トークン） | 毎セッション自動読み込み |
| MEMORY.md + 参照ファイル | 内容による | 自動読み込み |
| MCPツール定義 | ツール1つ ≒ 500トークン | 有効なMCPが多いほど増える |
| スキル定義（呼び出し時） | 内容による | スキルを呼ぶたびに読み込み |
| セッション要約（SessionStart hook） | 数百トークン | 前回セッションの要約 |

MCPサーバーを10個有効にするだけで約5,000トークンが常時消費されます。「全部入れとけば便利」は実は逆効果です。

## `/context` コマンドの3モード

### Budget モード（診断）

```
/context
```

または

```
/context budget
```

現在のコンテキスト状態を診断します。

**出力内容:**
- 総コンテキスト使用量（推定）
- 固定消費の内訳（CLAUDE.md、メモリ、MCPツール）
- 健康度: 🟢（余裕あり）/ 🟡（注意）/ 🔴（危険）
- 推奨アクション

---

### Compact モード（判断）

```
/context compact
```

または

```
/context 今compact?
```

「今 `/compact` すべきかどうか」を判断します。

**判断基準:**
- 会話の長さ（何ターン経過しているか）
- 現在作業中の重要度（途中で圧縮すると困るか）
- 直後に重い作業があるか

---

### Optimize モード（治療）

```
/context optimize
```

または

```
/context 重い
```

コンテキストを実際に削減するアクションを提案・実行します。

**典型的なアクション:**
- 使っていないMCPサーバーを無効化
- CLAUDE.mdの冗長な記述を圧縮
- 大きなメモリファイルを要約して縮小

---

## `/compact` の最適タイミング

### やるべきタイミング

- 1つのフェーズ（調査・設計・実装・テスト）が完了したとき
- 会話が30〜50ターンを超えてきたとき
- 回答の品質が落ちてきたと感じたとき
- 次のフェーズに進む前（重い作業の前）

### やらないべきタイミング

- ファイル編集の途中（変更内容が消えるわけではないが文脈は失われる）
- 複雑な条件を積み上げている最中（「A かつ B の場合だけ C」のような複合指示）
- 圧縮直後（二重圧縮しても効果が薄い）

### 圧縮前にメモを付ける

```
/compact これまでの進捗:
- ヒーローセクション完成
- ナビゲーションはモバイル対応が残っている
- 次: Aboutセクションの実装
```

このメモが圧縮後の要約に含まれるため、作業文脈を保ったまま再開できます。

## 何が残り、何が消えるか

`/compact` を実行すると:

| 残るもの | 消えるもの |
|---------|-----------|
| ファイルへの変更（実際のファイルは変わらない） | 会話の詳細な経緯 |
| CLAUDE.md（再読み込みされる） | 中間的な判断・議論の文脈 |
| メモリファイル | 「なぜその判断をしたか」の背景 |
| 圧縮時に書いたメモ | 読み込んだファイルの内容（再読み込みが必要）|

重要なことは「消えないファイルに書いておく」が基本です。

## 手動 vs 自動 compact

### 手動 compact（推奨）

自分で `/compact` を打つ方法です。タイミングをコントロールできるため推奨します。

### 自動 compact

`settings.json` で `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` を設定すると、コンテキストが指定割合を超えたときに自動圧縮されます。

```json
{
  "env": {
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50"
  }
}
```

デフォルトは95%（ほぼいっぱいになってから圧縮）です。50%に設定すると早めに圧縮されるため、品質低下を防ぎやすくなります。

**自動圧縮のデメリット:** タイミングを制御できないため、作業の途中で圧縮されることがあります。

## コンテキストを節約するコツ

### サブエージェントを使う

```
調査エージェントを使って〇〇を調べてきて。結果だけ要約で返して。
```

サブエージェントは別のコンテキストウィンドウで動作します。調査中の詳細はメインセッションには入らず、「結果の要約」だけが返ってきます。

### MCPを必要なものだけに絞る

使っていないMCPサーバーは `settings.json` から削除するか、コメントアウトしてください。1つのMCPサーバーが500〜3,000トークンを固定消費します。

### CLAUDE.mdを簡潔に保つ

CLAUDE.mdは毎セッション読み込まれます。長くなってきたら:
- 詳細は `memory/` ファイルに移動
- CLAUDE.mdはルールと参照先（ファイルパス）だけに絞る

### 大きなファイルは部分読み込みを指示する

```
# 悪い（ファイル全体を読み込む）
このファイルを読んで分析して

# 良い（必要な部分だけ読む）
このファイルの100〜200行目だけ読んで、関数名の一覧を教えて
```

## 品質低下のサイン

以下が見られたら、コンテキストが限界に近づいているサインです。

- **回答が短くなった:** 以前は詳しく説明していた内容を省略するようになった
- **指示を忘れる:** CLAUDE.md に書いたルールを守らなくなった
- **同じことを聞いてくる:** 「それは先ほどお伝えしましたが...」という状況
- **以前のファイルを把握していない:** 会話の序盤で触れたファイルを忘れる

---

## 対処法

### 軽度の場合: `/compact` にメモ付き

```
/compact
作業中ファイル: ~/project/index.html（ヒーローセクションまで完成）
次のタスク: Aboutセクションの実装
重要な決定: フォントはNoto Sans JP、アクセントカラーは #2563EB
```

### 重度の場合: 新しいセッション

品質が著しく低下している場合、`/compact` よりも新しいセッションを開く方が効果的です。重要な情報はメモリファイルに書いておき、新セッションで自動読み込みさせてください。

**判断基準:** 「/compact を3回以上繰り返しても品質が戻らない」なら新セッション。

---

## まとめ: コンテキスト管理の鉄則

1. **固定消費を最小化する:** MCPは必要最低限、CLAUDE.mdは簡潔に
2. **大事なことはファイルに書く:** 会話は消えるが、ファイルは残る
3. **フェーズの区切りで /compact する:** 作業の途中ではなく、段落の終わりで
4. **品質低下に気づいたらすぐ対応:** 我慢してもよくならない
5. **数字より体感が大事:** コンテキスト使用率が80%でも回答が鋭ければ問題なし。70%でも回答が雑なら対処する
